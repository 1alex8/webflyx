Inititalize Repo
    Command : git init
            Inititalizes a repo, must be executed in the working dir

Show hidden content of dir
    Command: ls -hidden / ls -a
            Shows all Dirs even the hidden ones 1 for Ps and 2 for bash

A File can be one of several states in a git Repo. Important ones:

            - untracked: Not being tracked by git
            - staged: Marked for inclusion in the next commit
            - committed : Saved to the repo's history
        
Status
    Command: git status 
            Shows you the current state of your repo. It will tell
            you which files are untracked, staged and committed

Add to Repo
    git add [path to file] || git add .
            first adds a single file to what will be committed; second adds everything
            use "git rm --cached <file>..." to unstage

Commit:
    git commit -m "your message here"

Trees and Blobs
    tree: Git's way of storing a directory
    blob: Gis's way of storing a file

Storing Data:
    Git stores entire Snapshots of files on a per-commit level. Not as thought: only the changes

Optimization: 
    It does have some performance Optimizations:
    - git compresses and packs files to store them more efficiently
    - git deduplicates files that are the same across diffrent commits.
      If a File doesn't change between commits, Git will only store it once



Config:
    git config --add --global user.name "Github/Codeberg username"
    git config --add --global user.email "Github/Codeberg E-mail"

        Take apart:
            git config: the command to interact with the .gitconfig File
            --add :      Flag stating you want to add a configuration
            --global:    Flag stating you want this cofig to be stored globally in your .gitconfig File The opposite is "local", which stores the configuration in the current repo only
            user:        The section
            name:        The key within the section
            "1alex8"     The value you want to set for the key
        
        we can also store anything in it, but you should only do it in the "local" .gitconfig
            Example:
            git config --add --local company.ceo "Alex"
            git config --add --local company.cto "Felix"
    
    git config --get <key>
            useful to get a single value: git config --get user.email
    
    git config --unset <key>
        it needs a specific section and key, then deletes that shit.
    git config --unset-all <key>
        it unsets all instances of that section.key for example if you have multiple instances of : company.ceo
        best example: if you have changed the init.defaultbranch a few times and want to clean up

    git config --remove-section <section>
        used to remove entire section out of the .gitconfig (don't forget to use local if you want to do that on your local gitconfig)
